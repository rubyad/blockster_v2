## 1. System Overview

The provably fair system uses a **commitment-reveal** cryptographic scheme to guarantee that:

- The server commits to a result **before** the player makes their bet
- The player can **independently verify** the result after settlement
- Neither party can manipulate the outcome after the commitment is made

### Architecture Layers

| Layer | Technology | Responsibility |
|-------|-----------|----------------|
| **Backend** | Elixir/Phoenix | Seed generation, result computation, game state management |
| **Storage** | Mnesia (distributed) | Game records with seeds, status, transaction hashes |
| **Blockchain** | Solidity (EVM) | Immutable commitment storage, bet placement, settlement verification |
| **Transaction Relay** | Node.js (BUX Minter) | Submits commitments and settlements on-chain via settler wallet |
| **Frontend** | LiveView + JS Hooks | Displays commitment before bet, verification modal after settlement |

### Three Seeds, One Result

```
server_seed  — Random 32 bytes, generated by server, kept secret until settlement
client_seed  — Deterministic SHA256 of bet details (user_id, amount, token, difficulty, predictions)
nonce        — Sequential integer per player, ensures uniqueness across identical bets
```

These three inputs produce a single **combined seed** that deterministically generates the game result.

## 2. The Commitment-Reveal Pattern

### Why It Works

The commitment-reveal pattern solves the fundamental trust problem in online gambling:

1. **Server can't cheat**: The commitment hash locks the server seed before the player bets. Changing the seed after would produce a different hash, which the player can detect.
2. **Player can't cheat**: The client seed is derived deterministically from bet parameters (not random client input), so the player cannot manipulate it.
3. **Both parties can verify**: After settlement, all inputs are revealed and anyone can recompute the result using standard SHA256.

### The Flow

```
BEFORE BET                     DURING BET                    AFTER BET
─────────────                  ──────────                    ─────────
Server generates               Player makes                  Server reveals
server_seed (secret)           predictions                   server_seed
      │                              │                            │
      ▼                              ▼                            ▼
commitment_hash =              client_seed =                 Player verifies:
SHA256(server_seed)            SHA256(bet_details)           SHA256(server_seed) == commitment
      │                              │                            │
      ▼                              ▼                            ▼
Posted ON-CHAIN                Combined with                 Recomputes results
(immutable)                    server_seed + nonce            independently
      │                              │
      ▼                              ▼
Shown to player                Result determined
BEFORE they bet                deterministically
```

## 3. Seed Generation & Cryptographic Algorithm

### Step 1: Server Seed (Random, Secret)

```elixir
server_seed = :crypto.strong_rand_bytes(32) |> Base.encode16(case: :lower)
# Result: 64-character lowercase hex string (e.g., "a1b2c3d4e5f6...")
```

- Generated using cryptographically secure random bytes
- Stored in Mnesia, **never transmitted** until after settlement
- This is the secret that determines all outcomes

### Step 2: Commitment Hash (Public, Immutable)

```elixir
commitment_hash = :crypto.hash(:sha256, server_seed) |> Base.encode16(case: :lower)
# With "0x" prefix for on-chain submission
```

- SHA256 of the **hex string** of the server seed (not the raw bytes)
- Submitted to the smart contract before the player places a bet
- Shown to the player in the UI as proof that the seed is locked

### Step 3: Client Seed (Deterministic from Bet Details)

```elixir
input = "#{user_id}:#{bet_amount}:#{token}:#{difficulty}:#{predictions_str}"
client_seed = :crypto.hash(:sha256, input) |> Base.encode16(case: :lower)
```

**Example input**: `"42:1000:BUX:2:heads,tails"`

**Why deterministic (not random)?**
- All inputs are visible in the game record — anyone can recompute it
- No trust required in the client's random number generator
- Makes verification fully transparent and reproducible
- No browser-side crypto libraries needed

### Step 4: Combined Seed

```elixir
combined_input = "#{server_seed}:#{client_seed}:#{nonce}"
combined_seed = :crypto.hash(:sha256, combined_input) |> Base.encode16(case: :lower)
```

- Concatenates all three values with `:` separators
- SHA256 produces 32 bytes (64 hex characters)
- This is the master seed from which all results are derived

### Step 5: Results from Combined Seed

```elixir
# Each byte of the combined seed = one coin flip
for i <- 0..(num_flips - 1) do
  byte = :binary.at(combined_seed_bytes, i)
  if byte < 128, do: :heads, else: :tails
end
```

- **byte < 128** (values 0–127) → heads (128/256 = exactly 50%)
- **byte >= 128** (values 128–255) → tails (128/256 = exactly 50%)
- Perfect 50/50 distribution with no bias

## 4. Commitment Hash Lifecycle

### Phase 1: Generation & On-Chain Submission

When a player loads the game page:

1. **Server generates** `server_seed` and `commitment_hash`
2. **Server submits** commitment to blockchain via transaction relay:
   ```
   POST /submit-commitment
   {
     "commitmentHash": "0xabc123...",
     "player": "0x456def...",
     "nonce": 5
   }
   ```
3. **Smart contract stores**: `commitments[hash] = {player, nonce, timestamp, used: false}`
4. **Mnesia stores**: Full game record with status `:committed`
5. **Frontend displays**: Commitment hash to the player (proof the seed is locked)

### Phase 2: Bet Placement

When the player places their bet:

1. **JavaScript hook** sends `placeBet(token, amount, difficulty, predictions, commitmentHash)` to the smart contract
2. **Smart contract validates**:
   - Commitment exists for this hash
   - Commitment is unused (prevents double-spending)
   - Commitment belongs to this player
3. **Contract marks** commitment as `used = true`
4. **Server calculates** results immediately (optimistic — for fast UI)
5. **Mnesia updates**: status → `:placed`, results stored

### Phase 3: Settlement & Reveal

After the animation completes:

1. **Server reveals** `server_seed` to the smart contract via settlement transaction
2. **Smart contract stores** the revealed seed for transparency
3. **Mnesia updates**: status → `:settled`, settlement_tx stored
4. **Player can now** open the Verify Fairness modal to see the server seed

### State Machine

```
:committed  →  :placed  →  :settled
     │              │
     └── :expired   └── :expired (if not settled within 1 hour)
```

## 5. Commitment Reuse Mechanism

### The Problem

If a player loads the game page, a commitment is submitted on-chain. If they leave without betting, the commitment sits unused. When they return, we need to either reuse it or create a new one.

### The Solution

```elixir
def get_or_init_game(user_id, wallet_address) do
  # 1. Calculate next expected nonce from placed/settled games
  next_nonce = max_nonce_from_placed_games + 1

  # 2. Check for existing pending/committed game
  case get_pending_game(user_id) do
    %{commitment_tx: tx, nonce: ^next_nonce} = existing when tx != nil ->
      # REUSE: Same player, correct nonce, confirmed on-chain
      {:ok, existing}

    _ ->
      # CREATE NEW: No reusable commitment found
      init_game_with_nonce(user_id, wallet_address, next_nonce)
  end
end
```

### Reuse Conditions (ALL must be true)

1. A pending/committed game exists for this user
2. The game has a confirmed `commitment_tx` (already on-chain)
3. The game's nonce matches the calculated next nonce
4. The commitment hasn't been used yet

### Nonce Calculation

```elixir
# Only count PLACED and SETTLED games (not abandoned committed ones)
next_nonce = games
  |> filter(status in [:placed, :settled])
  |> map(& &1.nonce)
  |> max()
  |> then(& &1 + 1)
```

This ensures abandoned commitments don't consume nonce slots.

|-----------|------|-------|---------------|------------|-----------|------------|
| -4 | Win One | 5 | Any 1 of 5 matches | 1.02x | 96.9% | ~1% |
| -3 | Win One | 4 | Any 1 of 4 matches | 1.05x | 93.75% | ~1% |
| -2 | Win One | 3 | Any 1 of 3 matches | 1.13x | 87.5% | ~1% |
| -1 | Win One | 2 | Any 1 of 2 matches | 1.32x | 75% | ~1% |
| 1 | Win All | 1 | 1 of 1 matches | 1.98x | 50% | ~1% |
| 2 | Win All | 2 | 2 of 2 match | 3.96x | 25% | ~1% |
| 3 | Win All | 3 | 3 of 3 match | 7.92x | 12.5% | ~1% |
| 4 | Win All | 4 | 4 of 4 match | 15.84x | 6.25% | ~1% |
| 5 | Win All | 5 | 5 of 5 match | 31.68x | 3.125% | ~1% |

Multipliers are stored as basis points (10000 = 1.0x) for integer precision.

### Win Determination

```elixir
# Win One (negative difficulty): ANY prediction matching its result = win
check_win(predictions, results, difficulty) when difficulty < 0 do
  Enum.zip(predictions, results) |> Enum.any?(fn {p, r} -> p == r end)
end

# Win All (positive difficulty): ALL predictions must match
check_win(predictions, results, _difficulty) do
  Enum.zip(predictions, results) |> Enum.all?(fn {p, r} -> p == r end)
end
```

### Payout Calculation

```elixir
payout = bet_amount * multiplier / 10000  # Only if won
profit = payout - bet_amount               # Net gain
```

## 7. Settlement Flow

### Automatic Settlement (Background Worker)

A GenServer (`BuxBoosterBetSettler`) runs every **1 minute** and settles any bets with status `:placed` that are older than **2 minutes**:

```elixir
# Find stuck bets
placed_games = games |> filter(status == :placed, age > 2 minutes)

# Settle each one
for game <- placed_games do
  settle_game(game.game_id)
end
```

### Settlement API Call

```
POST /settle-bet
{
  "commitmentHash": "0xabc123...",
  "serverSeed": "0xa1b2c3d4...",    // NOW REVEALED (with 0x prefix)
  "results": [0, 1, 0],             // 0=heads, 1=tails
  "won": true
}
```

### On-Chain Settlement

The smart contract's `settleBet` function:
1. Validates the bet exists and is pending
2. Checks the bet hasn't expired (1-hour window)
3. Stores the revealed `serverSeed` in the commitment struct
4. Processes payout (transfer tokens to winner, or keep for house)
5. Emits `BetSettled` and `BetDetails` events

## 8. Smart Contract Design

### Data Structures

```solidity
struct Commitment {
    address player;      // Who this commitment is for
    uint256 nonce;       // Expected nonce for this bet
    uint256 timestamp;   // When commitment was made
    bool used;           // Whether this commitment has been bet on
    bytes32 serverSeed;  // Empty until settlement reveals it
}

struct Bet {
    address player;
    address token;
    uint256 amount;
    int8 difficulty;
    uint8[] predictions;
    bytes32 commitmentHash;  // Links back to commitment
    uint256 nonce;
    uint256 timestamp;
    BetStatus status;        // Pending → Settled/Refunded
}

// Storage mappings
mapping(bytes32 => Commitment) public commitments;         // hash → commitment
mapping(address => mapping(uint256 => bytes32)) public playerCommitments;  // player → nonce → hash
mapping(bytes32 => Bet) public bets;                       // commitmentHash doubles as betId
```

### Key Functions

#### `submitCommitment(bytes32 commitmentHash, address player, uint256 nonce)`
- Called by settler before player bets
- Stores commitment on-chain (immutable)
- Emits `CommitmentSubmitted` event

#### `placeBet(address token, uint256 amount, int8 difficulty, uint8[] predictions, bytes32 commitmentHash)`
- Called by player's smart wallet
- Validates commitment exists, is unused, and belongs to caller
- Marks commitment as `used = true`
- Creates Bet record keyed by commitmentHash
- Transfers bet amount from player to contract

#### `settleBet(bytes32 commitmentHash, bytes32 serverSeed, uint8[] results, bool won)`
- Called by settler after result is computed
- Stores `serverSeed` in commitment (for on-chain transparency)
- Processes payout based on `won` flag
- Emits `BetSettled` with all details including revealed serverSeed

### View Functions (For Verification)

```solidity
// Get current unused commitment for a player
getPlayerCurrentCommitment(address player) → (hash, nonce, timestamp, used)

// Get commitment by nonce (for historical lookup)
getCommitmentByNonce(address player, uint256 nonce) → (hash, timestamp, used, serverSeed)

// Get full commitment details
getCommitment(bytes32 hash) → (player, nonce, timestamp, used, serverSeed)
```

### Design Decision: Server as Source of Truth

The smart contract **trusts the settler** for result computation. It does NOT recompute results on-chain because:

1. **Gas efficiency**: SHA256 + byte comparison on-chain is expensive
2. **All data is recorded**: Server seed, results, and win status are stored in events
3. **Anyone can verify off-chain**: The math is simple enough to verify with any SHA256 calculator
4. **The commitment prevents cheating**: The server can't change the seed after committing

## 11. Security Properties & Guarantees

### What The System Guarantees

| Property | How It's Achieved |
|----------|------------------|
| **Server can't change result after bet** | Commitment hash on-chain before bet; SHA256 is irreversible |
| **Player can't predict result before bet** | Server seed is secret until settlement |
| **Results are deterministic** | SHA256(server_seed:client_seed:nonce) always produces same output |
| **Client seed can't be manipulated** | Derived deterministically from public bet details |
| **Each game is unique** | Nonce ensures different combined seed even for identical bets |
| **50/50 fairness per flip** | Byte < 128 = heads (128 values), byte >= 128 = tails (128 values) |
| **Full transparency** | All seeds, hashes, and transaction hashes revealed after settlement |
| **Independent verification** | Anyone with SHA256 access can verify any game |

### Critical Implementation Rules

1. **NEVER reveal server seed before settlement** — the entire security model depends on this
2. **ALWAYS submit commitment on-chain BEFORE the player bets** — this is the trust anchor
3. **Client seed MUST be deterministic from public data** — no random client input that could be manipulated
4. **Nonces MUST be sequential and unique per player** — prevents replay attacks
5. **Settlement MUST record the server seed on-chain** — enables permanent auditability
6. **Smart contract MUST verify commitment exists and is unused** — prevents double-spending

### Trust Model

```
Server:    Trusted to not reveal seed early (enforced by code, not visible to player until settlement)
Contract:  Trusted to store commitments immutably (enforced by blockchain consensus)
Settler:   Trusted to submit correct results (can be audited via on-chain data + off-chain verification)
Player:    Zero trust required — can verify everything independently
```

## 13. Pseudocode: Complete Algorithm

This is a language-agnostic summary of the entire system, suitable for reimplementation:

```python
# ============================================================
# PHASE 1: COMMITMENT (Before player bets)
# ============================================================

def create_commitment(player_address, nonce):
    # Generate cryptographically secure random seed
    server_seed = random_bytes(32).to_hex()     # 64 hex chars

    # Hash the seed to create the commitment
    commitment_hash = sha256(server_seed)        # Hash of the HEX STRING, not raw bytes

    # Submit commitment to blockchain (immutable)
    tx_hash = smart_contract.submitCommitment(commitment_hash, player_address, nonce)

    # Store in database (server_seed is SECRET)
    db.store(game_id, server_seed, commitment_hash, nonce, status="committed")

    # Show commitment_hash to player (proof the seed is locked)
    return {commitment_hash, tx_hash}


# ============================================================
# PHASE 2: BET PLACEMENT (Player makes predictions)
# ============================================================

def place_bet(game_id, user_id, bet_amount, token, difficulty, predictions):
    game = db.get(game_id)

    # Player calls smart contract with the commitment hash
    smart_contract.placeBet(token, bet_amount, difficulty, predictions, game.commitment_hash)
    # Contract validates: commitment exists, unused, belongs to player
    # Contract marks commitment as used

    # Server calculates result immediately (optimistic)
    result = calculate_result(game.server_seed, user_id, bet_amount, token, difficulty, predictions, game.nonce)

    db.update(game_id, status="placed", results=result.flips, won=result.won)
    return result


# ============================================================
# PHASE 3: RESULT COMPUTATION (Deterministic)
# ============================================================

def calculate_result(server_seed, user_id, bet_amount, token, difficulty, predictions, nonce):
    # Step 1: Derive client seed from bet details (deterministic)
    client_seed_input = f"{user_id}:{bet_amount}:{token}:{difficulty}:{','.join(predictions)}"
    client_seed = sha256(client_seed_input)

    # Step 2: Combine all three inputs
    combined_input = f"{server_seed}:{client_seed}:{nonce}"
    combined_seed = sha256(combined_input)

    # Step 3: Extract results from combined seed bytes
    combined_bytes = hex_to_bytes(combined_seed)
    num_flips = get_flip_count(difficulty)

    flips = []
    for i in range(num_flips):
        if combined_bytes[i] < 128:
            flips.append("heads")
        else:
            flips.append("tails")

    # Step 4: Determine win/loss
    if difficulty < 0:  # Win One mode
        won = any(pred == flip for pred, flip in zip(predictions, flips))
    else:               # Win All mode
        won = all(pred == flip for pred, flip in zip(predictions, flips))

    # Step 5: Calculate payout
    multiplier = MULTIPLIERS[difficulty]
    payout = (bet_amount * multiplier / 10000) if won else 0

    return {flips, won, payout, client_seed, combined_seed}


# ============================================================
# PHASE 4: SETTLEMENT (Reveal server seed)
# ============================================================

def settle_game(game_id):
    game = db.get(game_id)

    # Reveal server seed to blockchain
    smart_contract.settleBet(
        game.commitment_hash,
        game.server_seed,     # NOW REVEALED
        game.results,
        game.won
    )
    # Contract stores server_seed for transparency
    # Contract processes payout

    db.update(game_id, status="settled")


# ============================================================
# PHASE 5: VERIFICATION (Anyone can do this)
# ============================================================

def verify_game(game):
    # Step 1: Verify commitment
    assert sha256(game.server_seed) == game.commitment_hash

    # Step 2: Recompute client seed
    input = f"{game.user_id}:{game.bet_amount}:{game.token}:{game.difficulty}:{game.predictions}"
    assert sha256(input) == game.client_seed

    # Step 3: Recompute combined seed
    assert sha256(f"{game.server_seed}:{game.client_seed}:{game.nonce}") == game.combined_seed

    # Step 4: Verify results
    bytes = hex_to_bytes(game.combined_seed)
    for i, flip in enumerate(game.results):
        expected = "heads" if bytes[i] < 128 else "tails"
        assert flip == expected

    return "VERIFIED: Game is provably fair"


# ============================================================
# COMMITMENT REUSE (Optimization)
# ============================================================

def get_or_create_game(user_id, wallet_address):
    next_nonce = max_nonce_from_settled_games(user_id) + 1

    existing = db.find_pending_game(user_id)
    if existing and existing.nonce == next_nonce and existing.commitment_tx:
        return existing  # Reuse existing commitment

    return create_commitment(wallet_address, next_nonce)
```

## Summary

The provably fair system rests on three pillars:

1. **Cryptographic commitment**: SHA256 is a one-way function. Once the server publishes `SHA256(server_seed)`, it cannot find a different seed that produces the same hash. The seed is locked.

2. **Deterministic derivation**: Every step from seeds to results uses deterministic functions (SHA256, byte comparison). Given the same inputs, anyone gets the same outputs. There's no randomness after the initial server seed generation.

3. **Temporal ordering**: The commitment is on-chain BEFORE the bet. The server seed is revealed AFTER the bet. This ordering ensures neither party can manipulate the outcome.

These three properties together mean: **the result was determined before the player bet, and the player can prove it.**
