## /play — Coin Flip

### 🔴 CF-01 — On-chain settlement fails with `InvalidServerSeed (6026)`

- **Repro**: place a bet (any difficulty), wait for the settlement retry loop
- **Symptom**: `CoinFlipBetSettler` HTTP 500 from settler with Anchor error `0x178a — Invalid server seed (hash does not match commitment)`
- **Observed bet**: `a54937db1fbc52c548f9d08ddf85b369` (0.05 SOL, difficulty Win All 1-flip / 1.98×)
- **Impact**: the SOL went into the bankroll on bet placement but the settle tx permanently fails → stake is stuck until the 5-min reclaim_expired window and the user manually signs a reclaim
- **Root cause theory**: server seed used to settle doesn't hash to the commitment posted at bet placement. Either the seed rotates between bet placement and settlement, or there's a race when two bets are placed close together (which is what I did — three quick bets). See `programs/blockster-bankroll/src/instructions/settle_bet.rs:132`
- **Log**: `Program 49up2uzZANpjTC3sgggbZazdHBii2vY9mVK3vk5dT2tm failed: custom program error: 0x178a`

#### Fix plan

- **Root cause hypothesis**: seed/commitment lookup is keyed on `{user_id, nonce}` or similar in Mnesia `:coin_flip_games` (see `lib/blockster_v2/coin_flip_game.ex:53-111 init_game_with_nonce/3` — seed written at line 93). When two bets fire within seconds, the settler at `lib/blockster_v2/coin_flip_bet_settler.ex` reads the latest row and submits *its* seed against the earlier bet's on-chain `commitment_hash`. SHA256(newer seed) ≠ older commitment → `settle_bet.rs:130-135` rejects with `InvalidServerSeed`. Program is correct; lookup is wrong.
- **Files likely to change**:
  - Backend: `lib/blockster_v2/coin_flip_game.ex` (key scheme for `:coin_flip_games`; `init_game_with_nonce/3`, `calculate_game_result/5`, any seed-lookup helper), `lib/blockster_v2/coin_flip_bet_settler.ex` (seed fetch for settlement), `lib/blockster_v2/mnesia_initializer.ex` (if table schema adds an index)
  - Contracts (settler): `contracts/blockster-settler/src/services/bankroll-service.ts` (`settleBet` — confirm pass-through only; add a defensive pre-submit `assert SHA256(seed) == commitment_hash`)
  - Contracts (program): `contracts/blockster-bankroll/programs/blockster-bankroll/src/instructions/settle_bet.rs` — **no change**, verification at 130-135 is correct
- **Proposed approach** (client/settler-only — no program upgrade):
  1. Reproduce on `bin/dev`: place 3 rapid bets, tail `:coin_flip_games` via `:mnesia.dirty_index_read`, confirm seed overwrite.
  2. Re-key the seed lookup by `bet_order_pubkey` (or `commitment_hash`) — whatever the on-chain `BetOrder` carries. Add an Mnesia index on the new field if needed.
  3. In `CoinFlipBetSettler`, fetch the seed by the pending bet's commitment, not by user+nonce.
  4. Add a settler-side assertion: recompute SHA256(seed) before submit; on mismatch, move bet to the manual-review queue (GLOBAL-02) and stop retrying.
  5. CLAUDE.md: `getSignatureStatuses` polling unchanged; no dependent-tx chaining introduced; on-chain remains source of truth.
- **Tests to add BEFORE shipping**:
  - `test/blockster_v2/coin_flip_game_test.exs` — **regression test that would have caught this**: "two consecutive `init_game_with_nonce/3` calls for the same user produce seeds that are both independently retrievable, and each is settleable against its own commitment."
  - `test/blockster_v2/coin_flip_bet_settler_test.exs` — "settler refuses to submit settle_bet when SHA256(seed) ≠ commitment_hash; routes bet to manual review."
  - `contracts/blockster-bankroll/tests/bankroll.ts` — Anchor program test: (a) settle_bet happy path with matching seed, (b) negative test with mismatched seed asserting `InvalidServerSeed (0x178a)`. Cover the verification path on-chain even though it's not changing — documents behaviour and catches accidental regressions on future program upgrades.
  - `test/blockster_v2_web/live/coin_flip_live_test.exs` — LV integration: "back-to-back bet placements both settle cleanly with PubSub confirmation."
- **Edge cases the plan must handle**:
  - Same user places two bets within <1s (original repro).
  - Settlement arrives out of order (bet 2 confirms before bet 1) — seed lookup still finds the correct bet's seed.
  - User reloads `/play` between placement and settlement — seed persists in replicated Mnesia across node1/node2.
  - GlobalSingleton failover between bet placement and settlement — settler on the new primary reads the same Mnesia row.
  - Integer balances on PubSub — format helpers elsewhere must still handle the new PubSub payloads (SOL pool updates, etc.).
- **Risk / blast radius**: if the new key scheme is wrong, *every* coin flip bet fails `InvalidServerSeed`, not just the race ones. Canary on node2 first, validate for 30 min before node1.
- **Effort estimate**: **L** (6–10 calendar hours — reproduce, fix, settler assertion, 4+ tests, canary).
- **Dependencies**: unblocks CF-02 (recovery is easier to reason about when settlement is deterministic); unblocks CF-07 (settlement PubSub wiring needs stable events); feeds GLOBAL-02 (manual-review queue hook).

### 🔴 CF-02 — No recovery path after failed settlement

- **Repro**: settlement fails (CF-01) → UI stays on the win/loss result card
- **Observed**: no "Try again" button, no "Next bet" button, no way to place a new bet
- **Only escape**: reload the page manually or wait for the 5-minute reclaim_expired banner
- **Impact**: user is locked out of /play until the 5-min timer fires

#### Fix plan

- **Root cause hypothesis**: `lib/blockster_v2_web/live/coin_flip_live.ex:2008-2019 handle_info({:settlement_failed, reason}, socket)` only transitions state on non-timeout errors (line 2017); on `:timeout` it falls through as a no-op. When CF-01 fires, settler retries forever (GLOBAL-02) so the LV never receives a terminal failure event — UI is stuck on the result card.
- **Files likely to change**:
  - Backend: `lib/blockster_v2_web/live/coin_flip_live.ex` (add `:settlement_timeout` assign + timer; expand `handle_info({:settlement_failed, _}, _)`; add `handle_event("reset_game", …)`)
  - Templates: same file (render block around lines 765+) — add "Place another bet" CTA whenever `@settlement_status in [:settled, :failed, :timeout, :manual_review]`
- **Proposed approach**:
  1. On render-card entry, start a 60s `Process.send_after(self(), :settlement_timeout, 60_000)` — if `{:settlement_complete, _}` lands first, cancel the timer.
  2. On `:settlement_timeout`, transition `@settlement_status = :timeout`, surface "Settlement still pending — check Recent games or try again."
  3. Add a `reset_game` event that returns state to `:idle`, reuses existing init path. Do NOT cancel subscriptions or clear `@recent_games`.
  4. If the settler (post-CF-01) surfaces `:manual_review`, render copy "This bet needs manual review. Contact support." plus reclaim CTA.
  5. Keep the 30s `:check_expired_bets` loop + 5-min reclaim_expired banner untouched — it's the last-resort safety.
- **Tests to add BEFORE shipping**:
  - `test/blockster_v2_web/live/coin_flip_live_test.exs` — **regression test that would have caught this**: "settlement failure with reason `:timeout` does NOT lock the user; 'Place another bet' CTA renders and returns state to `:idle`."
  - Same file: "result card renders 'manual review' copy when settlement_status == :manual_review."
  - Same file: "reset_game event does not cancel PubSub subscription for balance/settlement topics."
- **Edge cases the plan must handle**:
  - Settlement succeeds after timeout timer fires (race) — if `{:settlement_complete, sig}` arrives post-timeout, still show the confirmed win card.
  - User clicks "Place another bet" with a pending bet still in flight — disable button until bet placement succeeds or a short grace window (avoid double-submit).
  - User navigates away and back — timer must not replay; check current settlement state on mount.
  - Multiple in-flight bets — reclaim banner enumerates all, not just the first.
  - Web3Auth silent reconnect loop after reset (same-user reconnect) — handle as a no-op per [memory](../memory/MEMORY.md).
- **Risk / blast radius**: if reset clears state too aggressively, an in-flight confirmed settlement is missed in the UI — user sees "try again" but their win already landed. Keep `@recent_games` and subscriptions intact across reset.
- **Effort estimate**: **M** (3–5 hours).
- **Dependencies**: bundle with CF-01 in one PR — reset path masks real settler bugs otherwise.

- **Proposed approach**:
  1. Add `attempt_count` + `last_attempt_at` to the pending-bet state.
  2. Exponential backoff schedule: 10s → 30s → 90s → 270s → 810s → cap at 900s (15min).
  3. Classify errors: transient (RPC timeout, network) → keep retrying; terminal (`InvalidServerSeed`, `AccountNotFound`, `InstructionError`) → move to manual-review after 3 consecutive occurrences, stop retrying.
  4. New Mnesia table `:coin_flip_manual_review` — bets marked by classification.
  5. Admin page `/admin/stats/stuck-bets` lists manual-review bets with reclaim CTA.
  6. Alerting: on dead-letter, `Logger.error` + optional Slack webhook if `SLACK_ALERT_WEBHOOK` env set.
  7. Route bets into manual-review from CF-01's new settler-side SHA256 assertion (ties the fixes together).
- **Tests to add BEFORE shipping**:
  - `test/blockster_v2/coin_flip_bet_settler_test.exs` — **regression test that would have caught this**: "bet that fails with `InvalidServerSeed` 3 times moves to `:coin_flip_manual_review` and stops retrying."
  - Same file: "retry delays follow exponential schedule (10, 30, 90, 270, 810, cap 900)."
  - Same file: "transient error (RPC timeout) does NOT dead-letter; keeps retrying indefinitely."
- **Edge cases the plan must handle**:
  - Transient RPC flaps — classify as retryable, don't dead-letter.
  - Settler restart loses in-memory retry state — persist state to Mnesia so failover survives.
  - GlobalSingleton handoff between node1 and node2 — retry state must be readable on the new primary.
  - Manual review bet resolved by admin reclaim — remove from queue and don't attempt to settle again.
  - PubSub balance broadcasts must still fire (see CF-08) — dead-letter path doesn't break confirmation flows.
- **Risk / blast radius**: too-aggressive dead-lettering routes transient errors to manual review → admin fatigue. Err on the side of retry, classify cautiously.
- **Effort estimate**: **M** (3–5 hours).
- **Dependencies**: benefits from CF-01 (fewer seed-mismatch retries in the first place); feeds CF-02 (manual-review state surfaces in LV).

### Settler retry strategy

GLOBAL-02 captures the core change. Generalize beyond coin flip: the same retry / dead-letter pattern should apply to every settler-to-chain operation (bet settlement, BUX mint, payment-intent sweep, airdrop claim). Pull the retry state machine into a shared module.

- **Structure**: `lib/blockster_v2/settler_retry.ex` — generic exponential-backoff + dead-letter queue. Takes `{operation_id, error_classifier, retry_config}`.
- **Error classifier**: per-operation function returning `:retry | :terminal | :transient`. Coin flip: `InvalidServerSeed` / `AccountNotFound` → terminal. RPC timeouts / `BlockhashNotFound` → transient.
- **Backoff schedule**: `[10, 30, 90, 270, 810, 900]` seconds, cap at 15 min.
- **Dead-letter table**: Mnesia `:settler_dead_letters` keyed by operation. Admin review at `/admin/stats/dead-letters`.
- **Alerting**: Logger.error + optional `SLACK_ALERT_WEBHOOK` if set.
- **Callers**: start with `CoinFlipBetSettler`; migrate `BuxMinter`, `PaymentIntentWatcher`, `AirdropClaimService` over time.

Estimate: **L** (5–8 hours for the shared module + coin flip caller; incremental as other callers migrate).

  The audit's "Mnesia seed overwrite" hypothesis was wrong — `:coin_flip_games` is keyed by a unique `game_id` per call, so rows don't clobber. The real bug lives on-chain:
  - `submit_commitment.rs:59` stores the commitment in `player_state.pending_commitment` (a **single field per player**, not per-nonce).
  - `place_bet_sol.rs:144` copies `bet_order.commitment_hash = player_state.pending_commitment` and clears the field.

  - Program is correct (per audit Don't-do list); the audit's prescribed fix (look up the seed by the on-chain `bet_order.commitment_hash`) works regardless of the race because SHA256 is unique.
